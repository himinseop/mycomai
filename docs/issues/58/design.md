# Issue #58 LLM 위키 — 질문 로그 기반 토픽 페이지 자동 생성·서빙

GitHub Issue: https://github.com/himinseop/mycomai/issues/58

## 배경

사용자 질문은 소수 토픽(포인트/정산/쿠폰/입점/결제취소 등)에 집중된다. 현재 파이프라인은 질문마다 청크 검색 → LLM 생성을 반복하는데:

- 여러 문서에 흩어진 정보를 종합해야 하는 질문에 약함 (청크 단위 검색의 한계)
- 같은 토픽 질문에도 검색 결과에 따라 답변이 흔들림 (일관성 부족)
- 답이 좋았던 질문도 매번 다시 생성 (비용·지연 반복)

**Knowledge Hub 직접 응답이 이 문제의 수동 해법으로 이미 검증됨** — 담당자가 정리한 Q&A를 유사 질문에 원문 제공. LLM 위키는 이 "정리 페이지 작성"을 자동화하고, 검수를 거쳐 같은 서빙 경로에 태운다.

## 목표

- 질문 로그(chat_history)에서 수요 상위 토픽을 도출하고, 토픽별 정리 페이지를 LLM으로 자동 생성
- 생성 페이지는 관리자 검수를 거쳐 서빙 (검수 전 컨텍스트 부스트 → 승인 후 직접 응답)
- 데이터 수집 후 소스 변경을 감지해 자동 재생성 (신선도 유지)

## 비목표

- 전 코퍼스 지식그래프 구축 (→ #59 GraphRAG에서 다룸)
- 사용자 편집 가능한 위키 UI (읽기 전용, 편집은 재생성으로)
- Knowledge Hub(담당자 수동 Q&A) 대체 — 공존하며 Hub가 항상 우선

## 핵심 설계

### 1. 데이터 흐름

```
[토픽 선정]  chat_history 최근 질문 → LLM 클러스터링 → 토픽 후보(대표 질문 포함)
                → 관리자가 채택 (Phase 1은 반자동)

[페이지 생성] 토픽별 대표 질문 3~5개로 하이브리드 검색 (기존 retrieve_documents 재사용)
                → 상위 문서 20~30건 수집 → LLM 합성 (구조화 페이지 + 출처 인용 필수)
                → wiki_pages 저장 (status=draft)

[검수]       관리자 위키 탭에서 미리보기 → 승인(approved) / 폐기(disabled)

[서빙]       대표 질문 임베딩을 ChromaDB에 저장 (is_wiki=True — Hub 패턴 그대로)
                draft:    검색 컨텍스트 최상단 부스트만 (직접 반환 X)
                approved: 유사 질문 매칭 시 Hub 경로로 직접 응답 (RRF 부스트 재사용)

[갱신]       수집 파이프라인 종료 후: 토픽별 소스 문서 재검색 → source_hash 비교
                → 변경 시 재생성 (재생성물은 다시 draft로 — 재검수)
```

### 2. 할루시네이션 통제 (가장 중요한 설계 결정)

위키 페이지는 LLM 생성물이므로 Hub 원문과 신뢰 수준이 다르다:

| 상태 | 서빙 방식 | 근거 |
|---|---|---|
| `draft` (생성 직후) | 검색 컨텍스트 부스트만 — 답변은 기존 RAG가 생성 | 검수 전 원문 노출 금지 |
| `approved` (검수 후) | Hub 직접 응답 경로 (원문 그대로 + 출처) | 사람이 확인한 콘텐츠 |
| 소스 변경으로 재생성 | 다시 `draft`로 강등 | 변경분 재검수 |

생성 프롬프트 규칙: 모든 서술 문단에 `[출처: 문서제목]` 인용 필수, 소스에 없는 내용 작성 금지,
상충하는 문서가 있으면 "문서 간 상이 — 확인 필요" 명시 (판단하지 않음).

### 3. 데이터 모델 (app_data.db)

#### wiki_pages

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | INTEGER PK | |
| `topic` | TEXT UNIQUE | 토픽 슬러그 (예: `point-accrual`) |
| `title` | TEXT | 페이지 제목 (예: "포인트 적립 정책·절차") |
| `content` | TEXT | 마크다운 본문 (출처 인용 포함) |
| `questions_json` | TEXT | 대표 질문 목록 (임베딩 매칭용, 3~10개) |
| `source_doc_ids_json` | TEXT | 합성에 사용된 원본 문서 ID |
| `source_hash` | TEXT | 소스 문서 content_hash 집합의 해시 (변경 감지) |
| `status` | TEXT | draft / approved / disabled |
| `model` | TEXT | 생성 LLM |
| `created_at` / `updated_at` | TEXT | ISO8601 |

ChromaDB: 대표 질문별 1문서 임베딩 (`is_wiki=True`, `wiki_id`, 질문 텍스트만) — Hub의 질문 임베딩 패턴 재사용. `KNOWLEDGE_HUB_RRF_BOOST`와 별도의 `WIKI_RRF_BOOST`(기본 3.0, Hub 5.0보다 낮게 — Hub 우선).

### 4. 모듈 구조

```
src/company_llm_rag/wiki/
├── topic_miner.py     # 질문 로그 → LLM 클러스터링 → 토픽 후보
├── page_builder.py    # 토픽 → 문서 수집 → LLM 합성 → wiki_pages 저장
├── wiki_store.py      # wiki_pages CRUD + 질문 임베딩 등록/제거
└── freshness.py       # source_hash 비교 → 재생성 대상 판정
prompts/wiki/page.txt  # 페이지 합성 프롬프트
scripts/build_wiki.py  # 수동/배치 실행 진입점 (수집 후 훅)
```

서빙 통합 지점:
- `retrieval_module.py`: is_wiki 문서 RRF 부스트 (Hub 부스트 로직 옆에 추가)
- `rag_system.py`: `_try_hub_direct_answer` 확장 — approved 위키도 직접 응답 대상 (Hub가 항상 우선)
- `data_loader.py` 종료부: freshness 체크 훅

### 5. 토픽 선정 (Phase 1: 반자동)

1. 최근 90일 질문(비질문·테스트 제외)을 LLM에 배치로 넘겨 클러스터 + 라벨 + 대표 질문 추출
2. 클러스터 크기순 정렬 → 관리자 위키 탭에서 "페이지 생성" 버튼으로 채택
3. 파일럿: 상위 10개 토픽

질문 수가 적은 초기에는 관리자가 토픽을 직접 입력하는 경로도 제공.

### 6. 페이지 합성 프롬프트 (요지)

```
당신은 사내 지식베이스 편집자입니다. 주어진 원본 문서들만 근거로
"<토픽>" 정리 페이지를 작성하세요.

구성: ## 개요 / ## 정책·기준 / ## 절차 / ## 자주 묻는 질문 / ## 출처
규칙:
1. 원본 문서에 없는 내용을 쓰지 마세요. 모든 문단 끝에 [출처: 제목] 표기.
2. 문서 간 내용이 상충하면 양쪽을 병기하고 "문서 간 상이 — 담당자 확인 필요"를 명시.
3. 오래된 문서(created_at 오래됨)와 최신 문서가 충돌하면 최신 문서를 우선하되 그 사실을 표기.
4. 수치·기간·조건은 원문 그대로 인용 (변환·추정 금지).
```

### 7. 관리자 위키 탭

- 페이지 목록: 토픽/상태/소스 문서 수/갱신일
- 미리보기(마크다운 렌더) + 승인/폐기/재생성 버튼
- 토픽 후보 목록 (클러스터 크기 = 수요 크기) + "페이지 생성" 버튼
- 직접 토픽 입력 생성

## 단계별 계획

### Phase 1 — 생성 파이프라인 + 검수 + 부스트 서빙
1. `wiki_pages` 테이블 + wiki_store
2. topic_miner (LLM 클러스터링) + page_builder (검색→합성→저장)
3. 관리자 위키 탭 (후보/목록/미리보기/승인)
4. draft 부스트 서빙 (retrieval_module에 WIKI_RRF_BOOST)
5. 테스트: 합성 출처 인용 검증, 부스트 순위, 상태 전이

### Phase 2 — 직접 응답 + 자동 갱신
1. approved 직접 응답 (`_try_hub_direct_answer` 확장, Hub 우선순위 보장)
2. 수집 후 freshness 훅: source_hash 변경 → 재생성 → draft 강등 + 관리자 알림(Teams webhook 재사용)
3. 회귀셋: 토픽별 대표 질문으로 위키 도입 전/후 A/B (ab_generation.py 패턴)

### Phase 3 — 자동화 고도화
1. 신규 질문 유입 시 기존 토픽 매칭 실패율 기반 신규 토픽 자동 제안
2. 페이지 간 상호 링크, 커버리지 리포트 (질문 대비 위키 적중률)

## 검증 시나리오 (Phase 1 TC)

| # | 시나리오 | 기대 결과 |
|---|---|---|
| 1 | 질문 로그 클러스터링 | 상위 토픽에 실제 빈발 주제(포인트/쿠폰 등) 포함 |
| 2 | 페이지 생성 | 모든 문단에 출처 인용 존재, 소스 외 내용 없음(LLM judge 검증) |
| 3 | draft 상태 질의 | 직접 응답 안 됨, 검색 상위에 위키 컨텍스트 포함 |
| 4 | approve 후 유사 질문 | 위키 직접 응답 + 출처 표기 |
| 5 | Hub 문서와 경합 | Hub가 우선 (부스트 5.0 > 3.0) |
| 6 | 소스 문서 변경 후 재수집 | source_hash 변경 감지 → 재생성 → draft 강등 |
| 7 | disabled 전환 | 임베딩 제거로 매칭 대상에서 즉시 제외 |
| 8 | 상충 문서 토픽 | "문서 간 상이" 명시, 임의 판단 없음 |

## 비용·성능 추정

- 생성: 토픽당 문서 25건×평균 500자 ≈ 입력 ~15k 토큰 → gpt-4o-mini 기준 토픽당 수십 원. 상위 10개 파일럿 전체 ~수백 원
- 서빙: approved 직접 응답은 LLM 호출 1회(안내 멘트)로 단축 — 기존 12초 → Hub 수준(~3초)
- 갱신: 수집 주기당 변경 토픽만 재생성 (source_hash 스킵)

## 리스크

| 리스크 | 대응 |
|---|---|
| LLM 합성 오류가 승인 페이지로 유입 | 검수 게이트 + 재생성 시 draft 강등 + 출처 인용 강제 |
| 위키가 낡은 정보 제공 | freshness 훅 + 페이지에 "기준일" 표기 |
| Hub와 중복/충돌 | Hub 부스트 우선 + Hub 존재 토픽은 생성 대상에서 제외 옵션 |
| 토픽 과잉 생성 | 관리자 채택 게이트 (자동 생성은 Phase 3에서도 제안까지만) |
