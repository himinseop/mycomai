# Issue #37 질문 그룹 단위 세션 재정의 및 피드백/분석 구조 개선

GitHub Issue: https://github.com/himinseop/mycomai/issues/37

## 배경

현재 `session_id`는 브라우저 상태를 유지하기 위한 기술적 식별자로 사용되고 있지만, 실제 UX에서 사용자가 인식하는 단위는 "새 질문으로 시작한 카드 1개"입니다.

예를 들어 사용자가 첫 질문을 하고 같은 카드에서 이어서 질문하면, 이 흐름은 하나의 질문 그룹으로 인식됩니다. 반면 운영 데이터에서는 턴별 `record_id` 중심으로 저장/피드백/분석이 이뤄져 다음 문제가 생깁니다.

- 후속 질문이 같은 질문 흐름인지 운영 화면에서 직관적으로 보이지 않음
- 피드백이 마지막 턴 1건에만 귀속되어 질문 그룹 전체 만족도를 해석하기 어려움
- 결과분석이 단일 `record_id` 기준이라 대화 맥락이 약함
- 브라우저 세션 개념과 질문 그룹 개념이 섞여 데이터 의미가 불분명함

## 목표

- `session_id`의 의미를 "브라우저 세션"이 아니라 "질문 그룹 ID"로 재정의한다.
- 새 질문 시작 시 새로운 `session_id`를 생성한다.
- 같은 카드에서 이어지는 후속 질문은 같은 `session_id`를 유지한다.
- 피드백과 결과분석을 질문 그룹 기준으로 집계하고 조회할 수 있게 한다.
- 관리자 화면에서 턴별 조회와 질문 그룹별 조회를 모두 지원할 수 있는 기반을 만든다.

## 비목표

- 여러 브라우저 탭 간 상태 동기화
- 로그인 사용자 기반의 장기 대화 이어쓰기
- 서로 다른 질문 그룹 자동 병합
- 기존 운영 데이터에 대한 완전 자동 재분류

## 현재 구조 요약

### 프론트엔드

- 새 질문 시작 시 카드가 생성됨
- 카드마다 별도 `session_id`를 생성하고 후속 질문에 재사용함
- 사용자가 체감하는 대화 단위는 이미 카드 단위와 거의 일치함

### 백엔드

- `chat_history`는 턴별 Q/A 레코드 저장소임
- 피드백은 `chat_history.feedback` 한 컬럼에 저장됨
- 결과분석은 `record_id` 단건 기준으로 수행됨
- 관리자 화면은 턴 목록 중심임

## 핵심 설계 방향

### 1. `session_id` 의미 재정의

별도 `question_thread_id`를 추가하지 않고, 기존 `session_id` 자체를 질문 그룹 ID로 사용합니다.

- 새 질문: 새 `session_id`
- 같은 카드 후속 질문: 기존 `session_id`
- 다른 새 질문: 또 다른 새 `session_id`

이 방식은 현재 프론트 구조와 가장 잘 맞고 변경 범위도 작습니다.

### 2. 데이터 모델 보강

`chat_history`는 계속 턴 단위 저장소로 유지하되, 질문 그룹 분석이 가능하도록 메타데이터를 보강합니다.

권장 추가 컬럼:

- `parent_record_id INTEGER DEFAULT NULL`
- `turn_index INTEGER DEFAULT 1`
- `group_feedback INTEGER DEFAULT 0`
- `group_feedback_at TEXT DEFAULT NULL`

설명:

- `parent_record_id`: 이전 턴과의 연결
- `turn_index`: 질문 그룹 내 몇 번째 턴인지 표시
- `group_feedback`: 질문 그룹에 대한 최종 피드백 상태
- `group_feedback_at`: 그룹 피드백 입력 시각

## 저장 구조 설계

### 턴 저장 규칙

질문 그룹의 각 턴은 모두 같은 `session_id`를 공유합니다.

예시:

| id | session_id | turn_index | parent_record_id | question |
|---|---|---:|---:|---|
| 88 | `sess-xes7nve` | 1 | `NULL` | 쿠폰의 난수번호 발행은 어떻게 하는거야? |
| 89 | `sess-xes7nve` | 2 | 88 | 난수번호는 몇개까지 만들 수 있어? |

저장 규칙:

- 첫 질문 저장 시 `turn_index = 1`, `parent_record_id = NULL`
- 후속 질문 저장 시 직전 레코드를 `parent_record_id`로 연결
- 같은 `session_id` 내 마지막 `turn_index + 1`로 저장

### 그룹 피드백 규칙

피드백은 "질문 그룹 전체가 유용했는가"를 기본값으로 해석합니다.

저장 방식:

- 사용자가 카드 하단에서 👍/👎 선택
- 해당 `session_id`에 속한 모든 턴의 `group_feedback`를 동일 값으로 갱신
- 상세 운영을 위해 클릭 지점의 `record_id`는 별도로 유지 가능

권장 정책:

- `feedback` 컬럼은 턴 피드백 용도로 유지
- `group_feedback`를 질문 그룹 대표 피드백으로 추가

이렇게 하면 단건 응답 품질과 질문 그룹 전체 만족도를 분리해서 볼 수 있습니다.

### 결과분석 규칙

분석 단위는 `record_id`가 아니라 `session_id`가 됩니다.

입력 데이터:

- 첫 질문
- 전체 질문/답변 transcript
- 마지막 질문
- 마지막 답변
- 그룹 피드백

분석 저장 권장 방식:

- 우선은 같은 `session_id`의 마지막 턴 레코드에 대표 분석 결과 저장
- 추후 필요 시 그룹 전용 테이블로 승격

## API 변경안

### `/chat`

요청:

- 기존처럼 `session_id` optional 허용
- 없으면 새 질문 그룹으로 간주

응답:

- `session_id`
- `record_id`
- `turn_index`

### `/chat/stream`

완료 메타 이벤트에 아래 필드를 포함합니다.

- `session_id`
- `record_id`
- `turn_index`
- `is_group_root`

### `/feedback`

현재는 `record_id` 기준 저장입니다. 아래처럼 질문 그룹 기준을 포함하도록 확장합니다.

요청 예시:

```json
{
  "session_id": "sess-xes7nve",
  "record_id": 89,
  "rating": -1,
  "scope": "group"
}
```

처리 규칙:

- `scope=group`: 같은 `session_id` 전체에 그룹 피드백 반영
- `scope=record`: 현재 턴에만 피드백 반영

초기 UX는 `scope=group`만 노출하는 것을 권장합니다.

## 관리자 화면 설계

### 목록 뷰

운영 화면은 두 가지 뷰를 지원하는 것이 좋습니다.

- 턴 뷰: 기존과 동일한 개별 Q/A 레코드 확인
- 그룹 뷰: `session_id`별 묶음 조회

그룹 뷰 컬럼 제안:

- `session_id`
- `root_question`
- `turn_count`
- `latest_turn_at`
- `group_feedback`
- `analysis_status`

### 상세 뷰

그룹 상세에는 아래 정보가 필요합니다.

- 첫 질문

- 그룹 피드백
- 분석 결과
- 관련 문서 참조 정보

## 마이그레이션 전략

### 스키마 마이그레이션

- 신규 컬럼 추가는 idempotent 하게 수행
- 기존 데이터는 최대한 안전하게 유지

### 기존 데이터 처리

기존 데이터는 과거 저장 규칙을 완벽히 복원하기 어렵기 때문에 보수적으로 처리합니다.

- 기본값으로 각 `session_id`를 그대로 유지
- `turn_index`는 `created_at` 오름차순으로 재계산
- `parent_record_id`는 같은 `session_id` 내 직전 레코드로 채움
- `group_feedback`는 기존 `feedback`가 있는 마지막 턴 값을 우선 사용

주의:

- 기존 `session_id`가 이미 브라우저 세션 개념으로 사용된 구간이 있다면, 질문 그룹이 과대 묶일 수 있음
- 이 구간은 관리자 화면에서 해석 시 주의가 필요함

## 테스트 케이스

### TC-01 새 질문 시작 시 새 세션 생성

- 목적: 새 질문이 새로운 질문 그룹으로 생성되는지 확인
- 사전조건: 메인 화면 진입
- 절차:
  1. 하단 입력창에서 질문 A 전송
  2. 응답 완료 후 메타 응답 확인
- 기대결과:
  - `session_id`가 생성됨
  - `turn_index = 1`
  - 관리자 화면에서 해당 그룹의 `turn_count = 1`

### TC-02 후속 질문은 같은 세션 유지

- 목적: 같은 카드의 이어서 질문이 같은 질문 그룹으로 저장되는지 확인
- 사전조건: 질문 A가 완료된 카드 존재
- 절차:
  1. 같은 카드에서 후속 질문 B 전송
  2. 저장된 2개 레코드 확인
- 기대결과:
  - 두 레코드의 `session_id`가 같음
  - 두 번째 레코드의 `turn_index = 2`
  - 두 번째 레코드의 `parent_record_id = 첫 번째 레코드 id`

### TC-03 다른 새 질문은 다른 세션으로 분리

- 목적: 카드가 다르면 서로 다른 질문 그룹으로 분리되는지 확인
- 절차:
  1. 질문 A 카드 생성
  2. 별도 새 질문 C 시작
- 기대결과:
  - 질문 A 그룹과 질문 C 그룹의 `session_id`가 다름
  - 관리자 그룹 목록에서 별도 항목으로 조회됨

### TC-04 그룹 피드백 저장

- 목적: 카드 하단 피드백이 질문 그룹 단위로 저장되는지 확인
- 사전조건: 2턴 이상 포함된 질문 그룹 존재
- 절차:
  1. 카드 하단에서 👍 또는 👎 선택
  2. DB 및 관리자 화면 확인
- 기대결과:
  - 해당 `session_id`의 그룹 피드백이 저장됨
  - 그룹 뷰에서 피드백 상태가 표시됨
  - 필요 시 클릭한 `record_id`도 추적 가능함

### TC-05 그룹 피드백과 턴 피드백 분리

- 목적: 그룹 피드백 저장 시 기존 턴 피드백 해석이 깨지지 않는지 확인
- 절차:
  1. 그룹 피드백 저장
  2. 단건 상세와 그룹 상세 조회
- 기대결과:
  - 그룹 피드백이 전체 흐름 대표값으로 표시됨
  - 개별 레코드 상세는 기존 정보와 호환됨

### TC-06 그룹 분석 수행

- 목적: 그룹 분석이 전체 transcript 기반으로 수행되는지 확인
- 사전조건: 2턴 이상 그룹에 👎 피드백 저장
- 절차:
  1. 분석 트리거 실행
  2. 결과 보고서 확인
- 기대결과:
  - 분석 입력에 이전 턴 문맥이 포함됨
  - 결과가 마지막 턴 단건이 아니라 질문 그룹 전체 맥락을 설명함

### TC-07 관리자 그룹 목록 조회

- 목적: 그룹 기준 목록 조회가 가능한지 확인
- 절차:
  1. 관리자 화면 진입
  2. 그룹 뷰 선택
  3. 목록 확인
- 기대결과:
  - 한 질문 그룹이 한 행으로 보임
  - `turn_count`, `root_question`, `group_feedback`가 표시됨

### TC-08 관리자 그룹 상세 조회

- 목적: 질문 그룹 상세 정보가 충분히 제공되는지 확인
- 절차:
  1. 그룹 목록에서 항목 선택
  2. 상세 모달 또는 상세 페이지 확인
- 기대결과:
  - 전체 transcript가 시간순으로 표시됨
  - 그룹 분석 결과가 표시됨
  - 참조 문서와 응답 메타 정보 확인 가능함

### TC-09 기존 데이터 마이그레이션

- 목적: 운영 중 기존 데이터가 안전하게 읽히는지 확인
- 절차:
  1. 마이그레이션 적용
  2. 과거 이력 조회
- 기대결과:
  - 앱 시작 시 오류 없이 마이그레이션 완료
  - 과거 데이터에 `turn_index`와 `parent_record_id`가 채워짐
  - 조회/통계 기능이 깨지지 않음

### TC-10 Teams 문의와 그룹 세션 호환

- 목적: 질문 그룹 기반 세션 구조에서도 Teams 문의가 정상 동작하는지 확인
- 절차:
  1. 2턴 이상 질문 후 Teams 문의 실행
  2. 전송 내용 확인
- 기대결과:
  - 같은 질문 그룹의 transcript가 요약에 포함됨
  - 다른 카드의 질문 내용은 섞이지 않음

## 수동 검증 순서 제안

1. 새 질문 생성
2. 같은 카드에서 2회 후속 질문
3. 다른 새 질문 생성
4. 첫 번째 카드에 그룹 피드백 입력
5. 관리자 그룹 목록/상세 확인
6. 분석 결과 확인
7. Teams 문의 확인
8. 앱 재시작 후 이력 재조회

## 리스크

- 기존 `session_id` 의미가 혼재된 데이터는 100% 정확한 그룹 복원이 어려움
- 그룹 피드백과 턴 피드백을 동시에 운영하면 개념 충돌이 생길 수 있음
- 관리자 화면을 턴 뷰와 그룹 뷰로 분리하지 않으면 오히려 더 혼란스러울 수 있음

## 대응

- 신규 데이터부터 질문 그룹 기준을 엄격히 적용
- UI 문구에서 피드백 단위를 명확히 안내
- 관리자 화면에 뷰 전환을 명확히 제공
- 과거 데이터는 "legacy grouping"으로 표시하는 방안 검토

## 성공 기준

- 같은 카드의 후속 질문은 항상 같은 `session_id`로 저장된다.
- 새 질문은 항상 새로운 `session_id`로 시작한다.
- 그룹 피드백과 그룹 분석을 관리자 화면에서 일관되게 조회할 수 있다.
- 2턴 이상 질문 흐름의 문맥이 분석 보고서에 반영된다.
